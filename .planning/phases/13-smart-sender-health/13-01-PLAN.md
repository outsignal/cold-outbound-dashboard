Create `src/lib/linkedin/health-check.ts` with the core detection engine:

```typescript
export interface HealthCheckResult {
  senderId: string;
  senderName: string;
  workspaceSlug: string;
  previousStatus: string;
  newStatus: string;
  reason: string;
  detail: string;
  bouncePct: number | null;
  severity: "warning" | "critical";
  reassignedCount: number;
  workspacePaused: boolean;
}

export async function runSenderHealthCheck(): Promise<HealthCheckResult[]>
```

Implementation details for `runSenderHealthCheck()`:

**Step 1: Fetch all active senders** (status = "active" or "setup") with their workspace info.

**Step 2: Compute bounce rates from WebhookEvent**
- Query `WebhookEvent` for the last 24h, grouped by `senderEmail` for `EMAIL_SENT` and `BOUNCED` events.
- Build a map of `senderEmail -> { sent: number, bounced: number }`.
- Compute bounce rate: `bounced / sent`. Flag if > 0.05 AND sent >= 10 (minimum volume gate per research).
- Match senderEmail to Sender.emailAddress using case-insensitive comparison.

**Step 3: Check LinkedIn health signals from LinkedInDailyUsage**
- For each sender, check yesterday's `LinkedInDailyUsage` record.
- If `captchaDetected === true` -> flag as "blocked" with reason "captcha".
- If `restrictionNotice === true` -> flag as "blocked" with reason "restriction".

**Step 4: Check session status**
- If `sender.sessionStatus === "expired"` AND `sender.healthStatus !== "session_expired"` -> flag as "session_expired".

**Step 5: Check soft-flag auto-recovery**
- For senders with `healthStatus === "warning"` and `healthFlaggedAt`:
  - If `now - healthFlaggedAt >= 48 hours` (cooldown):
    - Recheck current bounce rate. If now <= 5%, auto-recover to "healthy".
    - Record SenderHealthEvent with reason "auto_recovered".
    - Clear `healthFlaggedAt` to null.

**Step 6: Apply flags and record events**
For each sender that needs flagging:
1. Determine severity: bounce_rate -> "warning"; captcha/restriction/session_expired -> "critical"
2. Update `sender.healthStatus` to the new status
3. Set `sender.healthFlaggedAt = new Date()` for soft flags (bounce_rate)
4. Create `SenderHealthEvent` record with status, reason, detail, bouncePct
5. If severity is "critical":
   - Check if this is the ONLY sender in the workspace (count active senders)
   - If only sender: pause all active campaigns in workspace using `prisma.$transaction`, set `workspacePaused = true`
   - Reassign pending LinkedIn actions from this sender to another healthy sender in the workspace using `prisma.linkedInAction.updateMany`. Use least-loaded selection: pick the healthy sender with fewest pending actions. Also check remaining daily budget from LinkedInDailyUsage. Set `reassignedCount` in result.
6. If severity is "warning" (bounce rate):
   - Do NOT reassign actions or pause campaigns for warning-level flags
   - Sender stays in rotation but is marked for monitoring

**Important edge cases:**
- Case-insensitive matching for senderEmail <-> Sender.emailAddress: use `.toLowerCase()` on both sides when building the map.
- Do NOT re-flag a sender that already has the same healthStatus (avoid duplicate SenderHealthEvents).
- Wrap the flag + campaign-pause in a `prisma.$transaction` for atomicity when pausing workspace campaigns.
- Return an array of `HealthCheckResult` for the caller to handle notifications.
