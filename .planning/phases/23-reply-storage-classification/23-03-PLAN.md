Create a GET endpoint following the webhook-log API pattern (paginated, filtered).

**Query parameters:**
- `page` (default 1), `limit` (default 50, max 100)
- `workspace` -- filter by workspaceSlug (optional, string)
- `intent` -- filter by intent value (optional, comma-separated for multi-select)
- `sentiment` -- filter by sentiment value (optional)
- `search` -- search senderEmail, senderName, subject, bodyText (optional, use `contains` with `mode: "insensitive"`)
- `campaignId` -- filter by campaignId (optional)
- `range` -- date range preset: "24h", "7d", "30d", "all" (default "all")

**Build Prisma where clause** from the query params. Use `AND` to combine filters. For `intent` filter, support comma-separated values with `in` operator. For `range`, compute a `receivedAt: { gte: cutoff }` filter.

**Response shape:**

**PATCH /api/replies/[id]/route.ts:**

Accept JSON body with optional fields: `overrideIntent`, `overrideSentiment`, `overrideObjSubtype`.

Validate that:
- `overrideIntent` is a valid Intent value (from INTENTS array) if provided
- `overrideSentiment` is a valid Sentiment value if provided
- `overrideObjSubtype` is a valid ObjectionSubtype value if provided (only allowed when effective intent would be "objection")

Update the Reply record:
```typescript
await prisma.reply.update({
  where: { id: params.id },
  data: {
    overrideIntent: body.overrideIntent ?? undefined,
    overrideSentiment: body.overrideSentiment ?? undefined,
    overrideObjSubtype: body.overrideObjSubtype ?? undefined,
    overriddenAt: new Date(),
    overriddenBy: "admin", // Single admin user, no auth system
  },
});
```

Return the updated Reply. Return 404 if Reply not found. Return 400 for invalid values.

**GET /api/replies/stats/route.ts:**

Accept query parameters: `workspace` (optional), `campaignId` (optional), `range` (optional, same presets as list endpoint).

Return aggregated stats:
```json
{
  "intentDistribution": [
    { "intent": "interested", "count": 42 },
    { "intent": "objection", "count": 18 },
    ...
  ],
  "sentimentDistribution": [
    { "sentiment": "positive", "count": 55 },
    { "sentiment": "neutral", "count": 30 },
    { "sentiment": "negative", "count": 15 }
  ],
  "workspaceCounts": [
    { "workspace": "rise", "count": 35 },
    { "workspace": "lime-recruitment", "count": 28 },
    ...
  ],
  "totalReplies": 100,
  "classifiedCount": 95,
  "unclassifiedCount": 5,
  "overriddenCount": 3
}
```

Use `prisma.reply.groupBy` for intent and sentiment distributions. Build where clause from filters (same pattern as list endpoint). For workspaceCounts, group by `workspaceSlug`.

Important: Use the "effective" values for distributions -- `COALESCE(overrideIntent, intent)` logic. Since Prisma groupBy doesn't support COALESCE, use `prisma.$queryRaw` with a raw SQL query for the intent/sentiment distributions:
```sql
SELECT COALESCE("overrideIntent", intent) as effective_intent, COUNT(*)::int as count
FROM "Reply"
WHERE intent IS NOT NULL
GROUP BY effective_intent
```

Or alternatively, use two separate queries (overridden vs not) and merge in JS -- whichever is simpler. The raw SQL approach is cleaner.
