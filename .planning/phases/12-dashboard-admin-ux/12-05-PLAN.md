Create GET `/api/linkedin-queue` endpoint:
- Query params:
  - `status` (optional): filter by action status
  - `actionType` (optional): filter by action type
  - `workspace` (optional): filter by workspaceSlug
  - `sender` (optional): filter by senderId
  - `page` (optional, default 1)
  - `limit` (optional, default 50, max 100)
- Returns:
  ```typescript
  {
    actions: Array<LinkedInAction & {
      sender: { id: string, name: string, workspaceSlug: string },
      // person info joined from Person table
      personEmail?: string,
      personName?: string
    }>,
    counts: {
      pending: number,
      running: number,
      complete: number,
      failed: number,
      cancelled: number,
      expired: number,
      total: number
    },
    total: number,
    page: number,
    totalPages: number
  }
  ```
- Always returns `counts` (unfiltered status counts for the header cards — filtered only by workspace/sender/actionType, not by status itself).
- The `actions` list is filtered by all params including status.
- Includes sender info via Prisma `include: { sender: { select: { id: true, name: true, workspaceSlug: true } } }`.
- Joins person info by querying Person where id = personId for each action (or use a raw query for efficiency). Actually, just include personId in the response and let the client decide — but to show person name in the table, do a batched lookup: collect unique personIds, query Person for those IDs, create a map, attach name+email to each action.
- Orders by: priority asc (highest first), scheduledFor asc (soonest first).
