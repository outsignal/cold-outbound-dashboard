  <action>
Create GET `/api/webhook-log` endpoint:
- Query params:
  - `search` (optional): free text search — searches leadEmail, senderEmail, and payload (using Prisma `contains` on leadEmail/senderEmail, and raw SQL `payload::text ILIKE '%search%'` for payload/subject search). For simplicity, use Prisma OR with `contains` on leadEmail and senderEmail. Payload search via raw SQL because Prisma doesn't support JSON text search well.
  - `eventType` (optional): filter by exact eventType
  - `workspace` (optional): filter by workspace slug
  - `errors` (optional, boolean): if true, filter to eventTypes that indicate errors — "BOUNCED", "UNSUBSCRIBED", "COMPLAINT"
  - `replies` (optional, boolean): if true, filter to "LEAD_REPLIED", "LEAD_INTERESTED", "UNTRACKED_REPLY_RECEIVED"
  - `hours` (optional): filter to events within last N hours (e.g., 24)
  - `page` (optional, default 1)
  - `limit` (optional, default 50, max 100)
- Filters combine (AND logic): search AND eventType AND workspace AND errors AND replies AND hours.
- Returns:
  ```typescript
  {
    events: WebhookEvent[],
    total: number,
    page: number,
    totalPages: number
  }
  ```
- Orders by receivedAt desc (most recent first).
- For search: if search param contains "@", prioritize leadEmail/senderEmail contains. Otherwise, search in payload text via `$queryRaw` for the payload search portion, combined with the Prisma where for other filters.
- Actually, keep it simple: for the initial implementation, search checks leadEmail contains OR senderEmail contains. Payload search is a nice-to-have but adds complexity. The executor can enhance later if needed.
  </action>
