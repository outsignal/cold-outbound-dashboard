# Phase 35: Email Inbox - Context

**Gathered:** 2026-03-11
**Status:** Ready for planning

<domain>
## Phase Boundary

Thread grouping API, conversation view, email reply composer, and AI suggested reply display in the client portal at /portal/inbox. Clients can read threaded email conversations and send replies via EmailBison. No LinkedIn inbox (Phase 36), no channel tabs or mobile layout (Phase 37).

</domain>

<decisions>
## Implementation Decisions

### Thread List View
- Sort by most recent activity first — threads with newest reply float to top
- Each thread row shows: contact name, email subject line, last message snippet (~80 chars), relative timestamp (e.g. "2h ago")
- Reply status indicator on each thread: awaiting reply (they replied, we haven't), replied (we sent last), new (unread)
- Interested threads get a subtle visual highlight (background color or badge) when the lead has the `interested` flag from EmailBison

### Conversation View
- Stacked email card layout — each message as a full card with email headers (From, To, Subject), stacked vertically
- Original outbound campaign email shown inline as the first message in the thread, same style as other outbound messages
- Inbound/outbound distinction: Claude's discretion on best visual approach (color-coded cards, alignment, or icon/label — whichever produces the best UX for a stacked email view)
- Render HTML email content (sandboxed for safety) — display emails as they were intended with formatting, links, images

### Reply Composer
- Fixed at the bottom of the conversation view — always visible, no need to click "Reply" first
- Auto-select sender email (the sender that received the reply) — no manual sender selection
- Wait for EmailBison API confirmation after hitting Send — show spinner, then add message to thread on success or show error on failure
- Plain text only — simple textarea, no rich text formatting

### AI Suggested Replies
- Show only when a suggestion exists on the reply record (generated by webhook handler) — no on-demand generation button
- Displayed as a dismissible card above the composer, between the last message and the composer area
- "Use this" button populates the composer textarea with the suggestion text — client reviews/edits, then hits Send manually
- Card is collapsible — minimizes to a single "AI suggestion available" line that can be expanded again

### Claude's Discretion
- Exact visual treatment for inbound vs outbound messages in stacked email view
- Thread list empty state design
- Loading states and skeletons
- Error state handling (failed to load threads, failed to send)
- Pagination or infinite scroll for thread list
- HTML email sandboxing approach

</decisions>

<specifics>
## Specific Ideas

- Wait-for-confirmation on send is chosen because email delivery to prospects has real consequences — optimistic UI is too risky for this context
- "Interested" highlight helps clients quickly spot hot leads without needing a separate filter
- Collapsible AI suggestion keeps the UI clean for clients who prefer to write their own replies while still surfacing the AI option

</specifics>
