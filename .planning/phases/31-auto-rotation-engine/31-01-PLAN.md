2. Create src/lib/domain-health/bounce-monitor.ts — the core state machine. This file contains ALL bounce monitoring logic:

   LOG_PREFIX = "[bounce-monitor]"

   **Constants:**
   - CONSECUTIVE_CHECKS_FOR_STEPDOWN = 6 (24h at 4-hour intervals)
   - EMAILBISON_MGMT_ENABLED check: `process.env.EMAILBISON_SENDER_MGMT_ENABLED === 'true'`

   **Type: EmailBounceStatus** = 'healthy' | 'elevated' | 'warning' | 'critical'

   **computeEmailBounceStatus(bounceRate: number | null, isBlacklisted: boolean): EmailBounceStatus | null**
   - Per CONTEXT.md locked decisions (NOT REQUIREMENTS.md thresholds):
     - isBlacklisted → 'critical'
     - bounceRate === null → return null (skip, no data)
     - bounceRate >= 0.05 → 'critical'
     - bounceRate >= 0.03 → 'warning'
     - bounceRate >= 0.02 → 'elevated'
     - else → 'healthy'

   **stepDownThreshold(currentStatus: EmailBounceStatus): number**
   - critical → 0.05 (must be below 5%)
   - warning → 0.03 (must be below 3%)
   - elevated → 0.02 (must be below 2%)
   - healthy → 0 (no step-down needed)

   **stepDown(status: EmailBounceStatus): EmailBounceStatus**
   - critical → warning
   - warning → elevated
   - elevated → healthy
   - healthy → healthy

   **statusSeverity(status: EmailBounceStatus): number**
   - healthy=0, elevated=1, warning=2, critical=3
   - Used to compare: if newSeverity > currentSeverity → escalation; if newSeverity < currentSeverity → potential step-down

   **evaluateSender(params)** — core per-sender evaluation logic:
   ```typescript
   async function evaluateSender(params: {
     sender: { id: string; emailAddress: string; workspaceSlug: string; emailBounceStatus: string; consecutiveHealthyChecks: number; emailBisonSenderId: number | null; originalDailyLimit: number | null };
     bounceRate: number | null;
     isBlacklisted: boolean;
   }): Promise<{
     transitioned: boolean;
     from?: string;
     to?: string;
     reason?: string;
     action?: string;
   }>
   ```
   Logic flow:
   a. Compute new status via computeEmailBounceStatus. If null (no data), skip — return { transitioned: false }.
   b. Get current status from sender.emailBounceStatus.
   c. If new status severity > current status severity → **ESCALATION**:
      - Reset consecutiveHealthyChecks to 0
      - Update sender: emailBounceStatus = newStatus, emailBounceStatusAt = now(), consecutiveHealthyChecks = 0
      - Create EmailHealthEvent: fromStatus, toStatus, reason (isBlacklisted ? 'blacklist' : 'bounce_rate'), bouncePct, detail string
      - If newStatus === 'warning' and EMAILBISON_MGMT_ENABLED:
        - Look up current daily limit from EmailBison (getSenderEmails or use cached data)
        - Store originalDailyLimit on sender if not already stored
        - Reduce daily limit by 50%: patchSenderEmail(senderEmailId, { daily_limit: Math.max(1, Math.floor(currentLimit / 2)) })
        - Set action = 'daily_limit_reduced'
      - If newStatus === 'critical' and EMAILBISON_MGMT_ENABLED:
        - Auto-remove sender from all active campaigns. Since the EmailBison campaign removal API is unknown (per research open question), implement as: log intent + set action = 'campaign_removal_pending'. When API is confirmed, add real removal. For now, the notification tells admin what to do manually.
        - Keep warmup active (do NOT disable warmup_enabled)
      - Return { transitioned: true, from: currentStatus, to: newStatus, reason, action }

   d. If new status severity === current status severity → **NO CHANGE**:
      - If currently elevated/warning/critical AND bounceRate is below stepDownThreshold(currentStatus):
        - Increment consecutiveHealthyChecks
        - If >= CONSECUTIVE_CHECKS_FOR_STEPDOWN (6):
          - Step down one level
          - Reset consecutiveHealthyChecks to 0
          - Update sender: emailBounceStatus = stepDown(currentStatus), emailBounceStatusAt = now()
          - Create EmailHealthEvent with reason: 'step_down'
          - If stepping down FROM warning and EMAILBISON_MGMT_ENABLED and originalDailyLimit is not null:
            - Restore daily limit: patchSenderEmail(senderEmailId, { daily_limit: originalDailyLimit })
            - Clear originalDailyLimit on sender
            - Set action = 'daily_limit_restored'
          - Return { transitioned: true, from: currentStatus, to: newStepDownStatus, reason: 'step_down' }
        - Else: just update consecutiveHealthyChecks on sender, return { transitioned: false }
      - If bounceRate >= stepDownThreshold(currentStatus):
        - Reset consecutiveHealthyChecks to 0 on sender
        - Return { transitioned: false }
      - If status is 'healthy': return { transitioned: false }

   e. If new status severity < current status severity → this means bounce rate dropped BUT we don't immediately step down. Just check step-down eligibility (same logic as d). The step-down is gradual per CONTEXT.md locked decisions.

   **runBounceMonitor()** — orchestrates the full cron run:
   ```typescript
   export async function runBounceMonitor(): Promise<{
     evaluated: number;
     transitioned: number;
     skipped: number;
     transitions: Array<{ senderEmail: string; from: string; to: string; reason: string }>;
   }>
   ```
   Logic:
   a. Query all senders with non-null emailAddress across all workspaces: `prisma.sender.findMany({ where: { emailAddress: { not: null }, status: { not: 'disabled' } }, select: { id, emailAddress, workspaceSlug, emailBounceStatus, consecutiveHealthyChecks, emailBisonSenderId, originalDailyLimit } })`
   b. For each sender, get latest BounceSnapshot: `prisma.bounceSnapshot.findFirst({ where: { senderEmail }, orderBy: { snapshotDate: 'desc' } })`
   c. For each sender, check blacklist status: `prisma.domainHealth.findUnique({ where: { domain: senderDomain } })` — isBlacklisted = overallHealth === 'critical'
   d. Batch the DB reads first (all snapshots + domain health), then process evaluations. This keeps the cron fast.
   e. Call evaluateSender for each sender. Collect results.
   f. Return summary with counts and transition details (these will be used by the cron route for logging and by notifications in Plan 02).

   NOTE: runBounceMonitor does NOT send notifications — that's Plan 02's responsibility. It returns the transition list so the cron route can pass them to the notification function.

   **replaceSender(params)** — sender replacement for critical senders:
   ```typescript
   export async function replaceSender(params: {
     criticalSender: { id: string; emailAddress: string; workspaceSlug: string };
   }): Promise<{ replacementEmail: string | null; reason: string }>
   ```
   Logic:
   - Find healthiest available sender in same workspace: `prisma.sender.findFirst({ where: { workspaceSlug, emailBounceStatus: 'healthy', status: 'active', emailAddress: { not: null }, id: { not: criticalSender.id } }, orderBy: ... })`
   - To find "lowest bounce rate", join with latest BounceSnapshot. Since Prisma can't do this natively, query all healthy active senders, then for each get latest bounceRate, pick lowest.
   - If found: return { replacementEmail, reason: 'Replaced with healthiest available sender' }
   - If no healthy senders available: return { replacementEmail: null, reason: 'No healthy senders available in workspace' }
   - NOTE: actual campaign reassignment depends on EmailBison API (unknown). This function identifies the replacement — the cron route includes the replacement info in the notification.
