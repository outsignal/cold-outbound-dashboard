  fromStatus: string;

Notification content per CONTEXT.md locked decisions:
- **Elevated → any**: informational with recommended action text. Message: "Sender {email} in {workspace} is now {toStatus}. Bounce rate: {bouncePct}%. Recommended: Monitor closely."
- **Warning escalation**: states what system did. "Sender {email} daily sending limit reduced by 50%. Current bounce rate: {bouncePct}%."
- **Critical escalation**: states what system already did. "Sender {email} removed from active campaigns. Bounce rate: {bouncePct}%. Warmup remains active." If replacementEmail: "Replacement sender available: {replacementEmail}." If no replacement: "No healthy replacement senders available — manual action required."
- **Step-down (recovery)**: informational. "Sender {email} recovered from {fromStatus} to {toStatus} after 24h sustained below threshold." If action === 'daily_limit_restored': "Daily sending limit restored to original value."
- **Blacklist-triggered critical**: "Sender {email} set to CRITICAL — domain is blacklisted."

Slack message format:
- Header block with status emoji + "Sender Health: {senderEmail}"
- Section block with mrkdwn: workspace, from→to status, bounce rate, reason, action taken
- If replacement available: additional section block with replacement info
- Context block with timestamp

Email format:
- Subject: "[Outsignal] Sender Health {toStatus}: {senderEmail}"
- HTML body: styled card with status badge (color-coded), sender details, action taken, bounce rate

Both Slack and email sends wrapped with audited() using notificationType: 'sender_health_{toStatus}' (e.g. 'sender_health_warning', 'sender_health_critical', 'sender_health_elevated', 'sender_health_healthy').

     d. Log summary: `${LOG_PREFIX} Evaluated ${result.evaluated}, transitioned ${result.transitioned}, skipped ${result.skipped}`
     e. Return JSON: { status: 'ok', evaluated, transitioned, skipped, transitions: result.transitions }

   - POST handler for admin manual override:
     a. Auth: require admin session. Import getServerSession (or whatever auth pattern other API routes in src/app/api/senders/ use). Check session exists and has admin role. Return 401 if not.
     b. Parse request body: { status: 'healthy' | 'elevated' | 'warning' | 'critical', reason?: string }
     c. Validate status is one of the four valid values. Return 400 if invalid.
     d. Get sender by id from URL params: prisma.sender.findUnique({ where: { id } })

     f. If sender has no emailAddress: return 400 "Sender has no email address".

        - reason: 'manual'
        - bouncePct: null (manual override)
        - detail: reason param or 'Manual override by admin'
        - senderEmail, senderDomain (derive from emailAddress), workspaceSlug
        - senderId: sender.id
     j. Update sender:
        - emailBounceStatus: requested status
        - emailBounceStatusAt: now()
        - consecutiveHealthyChecks: 0 (reset on any manual change)
     k. Return 200 with { success: true, from: currentStatus, to: status, eventId }
   - Per CONTEXT.md: "Next cron check resumes automatic evaluation after manual override (no locking)" — so no special lock field needed. The cron will simply evaluate the sender normally on next run.
