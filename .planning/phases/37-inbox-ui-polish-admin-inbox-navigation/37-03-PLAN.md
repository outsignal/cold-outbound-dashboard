  <action>
    Create 6 admin inbox API routes. These mirror the portal inbox routes but: (a) do NOT use `getPortalSession()` — admin routes have no portal session; (b) accept `workspace` query param or body field to scope queries; (c) when workspace is empty/missing, query across ALL workspaces; (d) include `workspaceName` and `workspaceSlug` in each thread/conversation response for the workspace badge.

    **1. GET /api/admin/inbox/email/threads:**
    - Read `?workspace=` from URL searchParams (optional).
    - Build Prisma where clause: if workspace provided, `{ workspaceSlug: workspace, direction: "inbound" }`. Otherwise `{ direction: "inbound" }`.
    - Query pattern: mirror the portal email threads route. Group replies by `emailBisonParentId`, aggregating to get latest reply per thread, count, interested flag, intent, sentiment, isRead, hasAiSuggestion.
    - For each thread, include `workspaceName` by joining to Workspace model: `const workspaces = await prisma.workspace.findMany({ select: { slug: true, name: true } }); const wsMap = new Map(workspaces.map(w => [w.slug, w.name]));` then `workspaceName: wsMap.get(thread.workspaceSlug)`.
    - Return `{ threads: ThreadSummary[] }` with `workspaceName` and `workspaceSlug` added.

    **2. GET /api/admin/inbox/email/threads/[threadId]:**
    - Parse threadId from params.
    - Query all replies in thread (same logic as portal variant).
    - Include cross-channel lookup (from Plan 02's pattern).
    - Include `aiSuggestedReply` from the latest inbound reply.
    - Return `{ messages, crossChannel, aiSuggestedReply, workspaceSlug, workspaceName }`.

    **3. POST /api/admin/inbox/email/reply:**
    - Body: `{ workspaceSlug, threadId, body, senderEmailId }`.
    - Validate `workspaceSlug` is provided.
    - Fetch workspace.apiToken: `const workspace = await prisma.workspace.findUnique({ where: { slug: workspaceSlug }, select: { apiToken: true, name: true } });`
    - If no apiToken, return 400 "Workspace not connected to EmailBison".
    - Create EmailBisonClient with apiToken, call sendReply (same pattern as portal reply route).
    - Return `{ success: true, replyId }`.

    **4. GET /api/admin/inbox/linkedin/conversations:**
    - Read `?workspace=` from URL searchParams (optional).
    - Query `prisma.linkedInConversation.findMany()` with optional `workspaceSlug` filter, ordered by `lastActivityAt desc`, limit 50.
    - Join Person records (two-query pattern from Phase 36-01): collect personIds, batch `prisma.person.findMany({ where: { id: { in: personIds } }, select: { id: true, firstName: true, lastName: true, linkedinTitle: true, companyDomain: true } })`.
    - Include `workspaceName` via workspace map.
    - Return `{ conversations: LinkedInConversationSummary[] }`.

    **5. GET /api/admin/inbox/linkedin/conversations/[id]/messages:**
    - Parse conversation id from params.
    - Query `prisma.linkedInMessage.findMany({ where: { conversationId: id }, orderBy: { sentAt: "asc" } })`.
    - Include cross-channel lookup: from conversation.personId, check `prisma.reply.findFirst()` for email thread.
    - Return `{ messages, crossChannel }`.

    **6. POST /api/admin/inbox/linkedin/reply:**
    - Body: `{ workspaceSlug, conversationId, body, personId }`.
    - Validate workspaceSlug.
    - Find an active sender for the workspace: `prisma.sender.findFirst({ where: { workspaceSlug, status: "active", sessionStatus: "active" }, select: { id: true } })`.
    - If no active sender, return 400 "No active LinkedIn sender for this workspace".
    - Create LinkedInAction (same pattern as portal LinkedIn reply route): `prisma.linkedInAction.create({ data: { type: "message", senderId, personId, workspaceSlug, payload: JSON.stringify({ conversationId, body }), priority: 1 } })`.
    - Return `{ success: true, actionId }`.
  </action>
